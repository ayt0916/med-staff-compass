## 2. アプリケーション概要

本アプリケーション「Med-Staff Compass」は、医療現場のスタッフが直面する「業務負荷」と「心理的ウェルビーイング」を可視化・統合分析するためのWebダッシュボードです。

### コンセプト
**「技術的効率化」を「スタッフの幸福度(Well-being)」に確実に結びつける、データ駆動型・現場伴走型プラットフォーム**

医療現場では、業務効率化（例：電子カルテ導入）が、必ずしもスタッフの負担軽減に繋がらず、むしろ「効率化で生まれた時間で、更なる肉体労働（患者ケア）をさせられる」という不信感が存在します。

本アプリは、この課題を解決するため、以下の2種類のデータを同時に可視化します。

1.  **物理的負荷:** 「管理的負荷（カルテ業務など）」と「身体的負荷（患者ケアなど）」のバランス
2.  **心理的負荷:** パルスサーベイに基づくウェルビーイング状態

これにより、経営層・管理者は「効率化が、スタッフの休憩時間確保や心理的安全性の向上に本当に繋がっているか」をデータで確認し、現場の不信感を払拭する（＝質的マンパワーの向上）ための具体的な意思決定を支援します。

---

## 3. 解決したい課題

* **現場の不信感:** 業務効率化によって削減された「管理的負荷」の時間が、単に「身体的負荷（患者ケア）」に振り分けられ、総労働負荷が変わらない（むしろ増える）ことへの不満。
* **人手不足の加速:** 上記の不信感や心理的ストレスが蓄積し、現場スタッフのエンゲージメントが低下。結果として離職に繋がり、人手不足がさらに加速する悪循環。
* **経営層の課題:** 経営層が「効率化＝負担軽減」となっていることをデータで確認できず、スタッフのメンタルヘルス状態も把握できないため、根本的な対策（人員配置の見直し、環境改善）が打てない。

## 4. 主な機能

本アプリケーションは、「スタッフ（個人）ビュー」と「管理者ビュー」の2つの視点を切り替えることができます。

### 共通機能
* **ビュー切り替え:** ヘッダーのトグルスイッチで、「スタッフ（個人）ビュー」と「管理者ビュー」をシームレスに切り替えます。

### スタッフ（個人）ビュー

スタッフ個人が自身の状態を客観的に把握するための機能です。

* **業務負荷バランス（個人）:**
    * 「身体的負荷」「管理的負担」「その他（休憩など）」の割合をドーナツチャートで可視化します。
    * **今月と先月のデータを比較表示**し、特に「管理的負担（削減目標）」と「その他（休憩など、確保目標）」の増減を数値で明示します。
* **心理的ウェルビーイング（個人）:**
    * 「ウィークリー・サーベイ」の結果をレーダーチャートで即時反映します。
* **ウィークリー・パルスサーベイ（入力）:**
    * モーダルウィンドウから「職場のサポート満足度」「業務の評価満足度」「ノンストレス度」の3項目を回答できます。

### 管理者ビュー

管理者（経営層・部門長）がチーム全体の傾向を把握し、意思決定を行うための機能です。

* **チーム全体の業務負荷バランス:**
    * スタッフ個人のデータを集計した、**チーム全体の「業務負荷バランス」**を今月・先月で比較表示します。
* **チーム全体のウェルビーイング:**
    * チームメンバーのサーベイ結果の**平均値**をレーダーチャートで可視化します。
* **チーム負荷ヒートマップ (モックアップ):**
    * （将来実装イメージ）「いつ」「誰の」負荷が高いかを可視化し、人員配置の偏りを特定するためのヒートマップです。
* **改善アクション (モックアップ):**
    * （将来実装イメージ）分析データに基づき、具体的な改善策を提案します。

## 5. 業務負荷バランスの算出ロジック（設計の根幹）

デモ画面のチャートは現在、静的なデモデータ（`script.js` 内の `staffData`, `adminData`）を用いて描画されています。

実際の現場でこの「身体的負荷」「管理的負荷」を算出・可視化するには、以下の3ステップが必要です。

### ステップ1： データの「入力」

まず、スタッフが「いつ、何の業務を、どれくらいの時間行ったか」を記録（＝タスクログ）する必要があります。実用的には、以下の2つのアプローチを組み合わせます。

#### アプローチ1： 手動入力（能動的ログ）
スタッフがスマートフォンや院内PCから、最小限の操作で「今から行う業務」をタップします。既存のシステムでは追跡できない業務（例：移動、雑務、突発的な対応）を記録するために不可欠です。

* **例：**
    * `[患者ケア]`ボタンをタップ → タイマースタート
    * ケア完了 → ストップ（この間の時間が「身体的負荷」として記録される）
    * `[カルテ記入]`ボタンをタップ → タイマースタート...

#### アプローチ2： 自動連携（受動的ログ）
「API連携」です。既存の院内システムのログ（いつ、誰が、何をしたか）を自動で収集します。

* **電子カルテ(HIS) / 看護支援システム(NSS):**
    * `10:05 田中看護師がAさんのカルテを開いた`
    * `10:10 田中看護師がAさんのカルテを保存した`
    * → この2つのログから「5分間のカルテ記入」があったと算出できます。

* **ナースコールシステム:**
    * `10:15 Bさんのナースコール発生`
    * `10:17 田中看護師がBさんの病室で対応開始`
    * `10:20 田中看護師が対応完了`
    * → これにより「2分間の移動」と「3分間の患者対応」が自動記録されます。

---

### ステップ2： データの「分類（ラベリング）」

ステップ1で収集した「タスクログ」（例：「カルテ記入」「患者対応」）を、「身体的負荷」「管理的負荷」「その他」のどれに該当するかを紐付け（マッピング）する定義マスターがアプリケーションの「肝」となります。

| 収集されたタスク（例） | → | 分類（カテゴリ） |
| :--- | :--- | :--- |
| `カルテ記入`（自動ログ） | → | **管理的負荷** |
| `サマリー作成`（手動入力） | → | **管理的負荷** |
| `カンファレンス`（手動入力） | → | **管理的負荷** |
| `バイタル測定`（手動入力） | → | **身体的負荷** |
| `ナースコール対応`（自動ログ） | → | **身体的負荷** |
| `点滴準備・実施`（手動入力） | → | **身体的負荷** |
| `休憩`（手動入力） | → | **その他** |
| `移動`（自動ログor手動） | → | **その他** |

---

### ステップ3： データの「集計と可視化」

最後に、分類されたデータをユーザーごと（この例では「田中看護師」）に、一定期間（例：「今月」）で集計します。

* **田中看護師の今月のタスク（例）:**
    * `管理的負荷` に分類されたタスクの合計時間： **80時間**
    * `身体的負荷` に分類されたタスクの合計時間： **200時間**
    * `その他` に分類されたタスクの合計時間： **120時間**
    * **総労働時間： 400時間**

この集計結果（80, 200, 120）を基に、パーセンテージ（`20%`, `50%`, `30%`）を算出し、`index.html` にあるようなドーナツチャートを描画します。

このように、実用的なアプリケーションでは、**「いかに現場の負担なくタスクログを自動収集し（ステップ1）」「それを病院の実態に合わせて正確に分類するか（ステップ2）」**が、システムの成功を左右する最も重要な設計ポイントとなります。
